1. The problem, in one paragraph

KYC onboarding is not a single request — it is a multi-step, long-running orchestration over slow, flaky external services. Naive implementations fail in seven predictable ways.

State corruption

Two async callbacks race on the same case. Audit log and status field diverge.

Silent data loss

A transient 5xx from OCR bubbles up as an unchecked exception; case stalls forever.

No bounded retry

Either one shot or infinite hammering. Both are bad.

No DLQ

Exhausted cases go to a status='ERROR' row nobody triages.

Reviewer vs. pipeline

Human approval races with an in-flight retry. Same status column, last write wins.

Bytes in the DB

ID card images blow up backups, complicate GDPR erasure.

No audit trail

"What happened to case X" requires grep across pod logs that already rolled over.

2. Solution shape

One central state machine. Bounded async executor. Per-step retry budget. First-class DLQ. Audit as a first-class artifact.

Every state change goes through KycStateMachine.fire(). Long steps run on workflowExecutor. Retries are scheduled durably (locally in the POC) with exponential backoff. Exhausted cases land in the DLQ with full context.

HTTP thread is released within milliseconds. All slow work happens on the executor. Retries are handed off to the scheduler; workflow threads stay free for real work.

5. Retry & DLQ mechanics

Exponential backoff with jitter

Formula from KycWorkflowService.computeBackoff:

backoff(n) = min( 500 * 2^n + jitter[0,500) , 10_000 )   // ms
AttemptBase (ms)With jitter (range)
1st retry500500 – 999
2nd retry10001000 – 1499
3rd retry20002000 – 2499
4th retry40004000 – 4499
≥ 5th retry—capped at 10 000

Jitter prevents a herd of retries hitting a recovering provider in lockstep. The cap keeps the wait bounded.

DLQ lifecycle

  1. Step fails, attempts++.
  2. If attempts < max → schedule next retry.
  3. If attempts >= max → DlqEntry(caseId, failedAtState, step, lastError, attempts) saved.
  4. Fire MOVE_TO_DLQ — case's audit log now records the DLQ transition.
  5. Operator lists DLQ, inspects, hits Replay.
  6. Fire REPLAY_FROM_DLQ → ID_UPLOADED, reset attempt counters, mark entry replayed, resubmit.

Idempotency: the DLQ entry has a replayed flag flipped inside the same call. Double-clicking Replay does not double-fire the pipeline.

7. Consistency when you scale beyond one JVM

The POC uses three mechanisms that only work inside one JVM. Deploy N replicas and each becomes an independent island. See CONSISTENCY.md for the full write-up.

Left: the POC deployed with N replicas — state is split across pod heaps, the synchronized block does nothing across JVMs, scheduled retries die with the pod. Right: after moving state to Oracle, replacing the monitor with a CAS on @Version, and swapping the local scheduler for a DB-polled timer with SELECT … FOR UPDATE SKIP LOCKED.

The three moves, in order

  1. Externalize state — JPA + Oracle with a @Version column.
  2. Externalize the lock — optimistic CAS on version, re-check legality on retry. (Pessimistic SELECT … FOR UPDATE and partitioned single-writer are alternatives.)
  3. Externalize the scheduler — a kyc_scheduled_work table polled with SKIP LOCKED, or a durable engine (Temporal / Quartz-JDBC).

Ordering matters. Skipping move 2 after move 1 (sharing state without a cross-pod lock) is the classic way to ship a subtly broken workflow.
